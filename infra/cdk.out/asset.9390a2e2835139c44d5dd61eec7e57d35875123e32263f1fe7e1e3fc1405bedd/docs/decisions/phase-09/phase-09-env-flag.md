# Decision — `--env` Flag and Environment Targeting

> Defines how all CLI commands target different environments via dotenv files.

---

## The Problem

Manguito CMS deployments typically involve multiple environments — local development, staging, production. Each environment has its own `DB_URL` and potentially other environment-specific config. Developers need a consistent, explicit way to target the right environment when running CLI commands.

---

## Approach

All CLI commands accept an `--env <file>` flag that loads a dotenv file before executing. This wraps dotenv-cli, which is already a declared project dependency from Phase 1.

```
manguito migrate --env .env.staging
manguito migrate --env .env.production
manguito start --env .env.production
manguito dev --env .env.local
manguito createsuperuser --env .env.staging
```

---

## Behavior

When `--env <file>` is provided:

1. The specified dotenv file is loaded first, before any other command logic executes.
2. Variables from the file are injected into `process.env` for the duration of the command.
3. If the file does not exist, the command stops immediately:

```
✖ Env file not found: .env.staging
  Check the file path and try again.
```

When `--env` is not provided, the command uses whatever environment variables are already present in the shell — no dotenv file is loaded automatically. This means a plain `manguito migrate` works correctly in CI/CD environments where variables are injected by the pipeline.

---

## Applies to All Commands

The `--env` flag is consistent across the entire CLI — a developer should never have to remember which commands accept it and which don't:

| Command | `--env` supported |
| ------- | ----------------- |
| `manguito dev` | ✔ |
| `manguito build` | ✔ |
| `manguito start` | ✔ |
| `manguito migrate` | ✔ |
| `manguito createsuperuser` | ✔ |
| `manguito validate` | ✔ |
| `manguito init` | ✔ |

---

## No Named Environment Profiles

There is no `--env staging` shorthand that maps to a predefined file. The developer always provides an explicit file path. Implicit environment name resolution adds magic that makes it harder to audit which file is actually being loaded. Explicit paths are always unambiguous.

---

## Dotenv File Conventions

Manguito CMS does not enforce a specific naming convention, but the following is recommended in documentation:

```
.env                  # local defaults (gitignored)
.env.local            # local overrides (gitignored)
.env.staging          # staging environment (gitignored)
.env.production       # production environment (gitignored)
.env.test             # test environment (may be committed without secrets)
```

The project scaffold generated by `manguito init` includes a `.gitignore` entry covering common env file patterns.

---

## Relationship to `manguito.config.ts`

The `--env` flag loads environment variables before `manguito.config.ts` is resolved. Config values that reference `process.env` correctly pick up values from the specified dotenv file:

```ts
// manguito.config.ts
export default defineConfig({
  db: {
    url: process.env.DB_URL!,  // resolved after --env file is loaded
  },
})
```

---

## CI/CD Usage

In CI/CD pipelines where environment variables are injected by the pipeline itself, `--env` is not needed — variables are already in `process.env`. For pipelines that use `.env` files checked into a secrets store, pass the file path explicitly:

```yaml
# example GitHub Actions step
- run: pnpm manguito migrate --force --env .env.production
```
