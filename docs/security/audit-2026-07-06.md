# Manguito CMS — Security Audit & Vulnerability Analysis

**Date:** 2026-07-06
**Branch:** `feature/mvp-security`
**Scope:** All packages (`core`, `db`, `api`, `admin`, `cli`), the generated server entries, and `infra/` (CDK). Static review only — no dynamic exploitation was performed.

This document is a **findings report**, the discovery half of the security work. Each finding names a location, an impact, and a remediation direction. It is intended to feed a separate remediation plan (via the writing-plans workflow), not to be a set of applied fixes.

## Severity legend

- **High** — exploitable with meaningful impact (data disclosure or script execution), even if it requires an authenticated low-privilege account.
- **Medium** — real weakness, but gated by preconditions (specific deployment mode, misconfiguration, or browser-side mitigations reduce practical impact).
- **Low / Informational** — hardening opportunities, defense-in-depth gaps, or issues confined to dev/build tooling.

## Findings

### 1. [High] Path traversal → arbitrary file read in the admin SPA static handler

**Location:** `packages/cli/src/codegen/server-entries.ts:128-143` (node `serverEntry`), and the shared Lambda/Vercel admin handler around `:193-205`.

The `/uploads/` handler defends against traversal (strips `..`, `resolve`, then a containment check `if (!filePath.startsWith(uploadsDir + '/'))`). The admin SPA handler does **not**:

```ts
const rel = path.slice(ADMIN_PREFIX.length) || '/'
const candidate = resolve(adminDir, '.' + rel)   // no containment check
// ... stat(candidate); readFile(filePath); serve with content-type
```

Node's HTTP server does not normalize `..` in the request path. A request like `GET /admin/../../../../etc/passwd` satisfies `path.startsWith(ADMIN_PREFIX)`, yields `rel = /../../../../etc/passwd`, and `resolve(adminDir, './../../../../etc/passwd')` escapes `adminDir`, reading an arbitrary file that is then returned in the response.

**Impact:** Unauthenticated arbitrary file read on the self-hosted node deployment (source, `.env` if colocated, secrets, `/etc/passwd`). The Lambda/Vercel handler shares the same pattern.

**Remediation:** Mirror the `/uploads/` guard: after `resolve`, reject unless `candidate === adminDir || candidate.startsWith(adminDir + sep)`. Strip `..` segments and/or use `path.normalize` + containment. Add a regression test hitting `/admin/../../etc/passwd` expecting 403/404.

### 2. [High] Stored XSS via same-origin serving of attacker-controlled upload extensions

**Location:** served content-type in `packages/cli/src/codegen/server-entries.ts:110-124` (`/uploads/` MIME map at `:56-71` includes `html` and `svg`); storage key extension derived from filename in `packages/api/src/storage/adapters/{s3,local}.ts:44-49 / 38-41`; MIME validation in `packages/api/src/routes/admin/media.ts:232-293`.

The accepted-MIME sets correctly exclude HTML/SVG (`IMAGE/VIDEO/FILE_MIME_TYPES`, media.ts:10-19). **But** the stored object key's extension is taken from the client `filename` (`path.extname(opts.filename)`) and is **never validated against the declared `mime_type`.** When the node server serves `/uploads/*`, the response `Content-Type` is computed from that extension via the MIME map, which maps `html → text/html; charset=utf-8` and `svg → image/svg+xml`.

Attack (requires a `media:create` account — e.g. writer):
1. Request a presigned URL with `mime_type=image/png` (passes validation) and `filename=payload.html`. Key becomes `image/<uuid>.html`.
2. Upload arbitrary bytes (`<script>…</script>`) — a presigned S3 PUT does not constrain body content; the direct-upload path also stores the raw bytes.
3. The file is served from `/uploads/image/<uuid>.html` as `text/html` **same-origin as the admin panel**. `X-Content-Type-Options: nosniff` does not prevent rendering when the type is explicitly `text/html`.

**Impact:** Stored XSS in the admin origin; a low-privilege user can execute script in an admin's session (privilege escalation, cookie access is blocked by `httpOnly` but DOM/API actions are not). For S3-served deployments the file is served from the bucket domain (different origin) — reduced but still hostile-content hosting.

**Remediation:** Decouple served type from filename: derive/allow-list the extension from the validated `mime_type` (canonical ext per MIME), or store with a safe extension. Serve all user uploads with `Content-Disposition: attachment` (or from a separate cookieless domain). Remove `html`/`svg` from the served MIME map for the uploads path, or force `application/octet-stream` for anything not on an image/video/pdf allow-list.

### 3. [Medium] User enumeration via login timing side-channel

**Location:** `packages/api/src/routes/admin/auth.ts:102-109`.

```ts
if (!user) { return c.json(invalidCredentials, 401) }        // fast path — no bcrypt
const passwordValid = await verifyPassword(password, user.password_hash)  // slow path — bcrypt cost 12
```

When the email does not exist, the handler returns before running bcrypt; an existing email runs a cost-12 compare (tens of ms). The error envelope is uniform, but the response-time difference lets an attacker enumerate valid accounts.

**Remediation:** On the not-found path, run `verifyPassword` against a fixed dummy bcrypt hash to equalize timing, then return the same error.

### 4. [Medium] CORS: unconditional credentials + hardcoded wildcard origin

**Location:** `packages/api/src/middleware/cors.ts` (sets `Access-Control-Allow-Credentials: true` unconditionally); `packages/api/src/app.ts:71` hardcodes `createCorsMiddleware({ origin: '*', enabled: true })`; infra default `ALLOWED_ORIGIN ?? '*'` in `infra/bin/infra.ts:41`.

Two problems: (a) the app-level CORS ignores the user-configured `server.cors.origin` and always uses `*`; (b) `Allow-Credentials: true` is set for every response. Browsers reject `*` + credentials, and the auth cookie is `SameSite=Strict` (strong CSRF mitigation), so this is not directly exploitable today — but if an origin is ever configured (the middleware then emits `Allow-Origin: <origin>` + `Allow-Credentials: true`), it becomes a valid credentialed grant with no per-request `Origin` validation, and a multi-origin config is joined with commas (invalid per spec).

**Remediation:** Have the API honor the configured origin; validate the request `Origin` against an allow-list and reflect a single matching origin; only send `Allow-Credentials: true` when a concrete (non-`*`) origin is allowed. Don't default to `*`.

### 5. [Medium] Login rate limiter keyed on `ip:email` enables password spraying

**Location:** `packages/api/src/routes/admin/auth.ts:23-32, 81` (limiter key `` `${ip}:${email}` ``, 10 attempts / 15 min).

Because each `(ip, email)` pair is an independent bucket, a single IP can try one password against many different emails without ever tripping the limit — the classic password-spraying gap. It does defend a single account against brute force.

**Remediation:** Add a second per-IP counter (attempts across all emails) alongside the per-account one; consider global backoff. Keep the per-account bucket too.

### 6. [Medium] No-op auth/permission shims used as default parameters (latent bypass)

**Location:** `packages/api/src/middleware/auth.ts:89-92` (`requireAuth`/`requirePermission` return `next()` with no checks); `packages/api/src/routes/admin/content.ts:172` uses the shim as the **default value** of the `requirePermission` parameter; route-level `requireAuth` sprinkled through `content.ts`/`media.ts`.

Today this is **not an active bypass**: the real `authMiddleware` is blanket-applied to `/admin/api/*` (`app.ts:150`) and the real `requirePermission` is passed explicitly at every call site (`app.ts:214-216`). The risk is latent: `registerAdminContentRoutes` silently falls back to a **no-op permission check** if ever called without the real middleware (a future codegen path, a test, a refactor), and the route-level `requireAuth` gives false assurance of protection.

**Remediation:** Delete the no-op shims. Make `requirePermission` a required parameter (no default). Remove the redundant route-level `requireAuth` or point it at the real middleware so it fails closed.

### 7. [Medium] Secrets injected as plaintext environment into Lambda/Fargate

**Location:** `infra/lib/cms-env.ts` (self-documented: "production deployments should graduate to Secrets Manager"); `infra/lib/lambda-stack.ts:55-57`; `infra/lib/fargate-stack.ts:66-68`.

`DB_URL` and `AUTH_SECRET` are passed via the `environment:` block. CDK bakes these into the CloudFormation template as plaintext, where they are readable by anyone with `cloudformation:GetTemplate`/`DescribeStacks` or console access to the function/task definition. (No secrets are committed to the repo — values come from the deployer's `process.env` at synth; `.env*` are gitignored and untracked. That part is clean.)

**Remediation:** Store `AUTH_SECRET` (and DB credentials) in AWS Secrets Manager / SSM SecureString and inject by reference (`ecs.Secret` / Lambda + extension), so plaintext never lands in the template. `AUTH_SECRET` disclosure is critical — it forges any session token.

### 8. [Medium] Presigned upload content and Content-Type not constrained to declared MIME

**Location:** `packages/api/src/routes/admin/media.ts:232-327`; `packages/api/src/storage/adapters/s3.ts:43-62`.

The presign step validates the declared `mime_type` and pins `ContentType` on the `PutObjectCommand`, but a presigned S3 PUT does not enforce that the client sends matching bytes, and there is no server-side re-inspection of the confirmed object's magic bytes. This is the storage-side enabler for finding #2 and also allows storing content that mismatches its recorded type.

**Remediation:** Enforce content-type via a signed policy/condition where the storage supports it; on `confirm`, re-fetch object metadata (or sniff magic bytes) and reject mismatches; canonicalize the served extension from the validated type (see #2).

### 9. [Low] No security headers on admin SPA or API responses

**Location:** generated server (`server-entries.ts`), API app (`app.ts`).

No `Content-Security-Policy`, `X-Frame-Options`/`frame-ancestors`, `Referrer-Policy`, or `Strict-Transport-Security` are emitted. A CSP in particular would materially reduce the blast radius of finding #2, and `X-Frame-Options` prevents clickjacking of the admin.

**Remediation:** Add a security-headers middleware (CSP with a strict default-src, `frame-ancestors 'none'`, `Referrer-Policy: no-referrer`, HSTS on HTTPS). Serve `X-Content-Type-Options: nosniff` on all responses, not just uploads.

### 10. [Low] Known advisories in the build/dev toolchain

**Evidence:** `pnpm audit --prod` reports 26 advisories (5 high / 19 moderate / 2 low), all in the build/dev chain — `esbuild` via `tsup` (`packages__cli>tsup>esbuild`, GHSA-g7r4-m6w7-qqqr) and via `drizzle-kit>@esbuild-kit` (GHSA-67mh-4wv8-2f99). These are dev-server CORS issues in tooling, not shipped in the runtime.

**Remediation:** Bump `tsup` and `drizzle-kit` to versions pulling `esbuild >= 0.28.1`. Low production risk, but they inflate the audit and the esbuild dev-server CORS issue matters if `manguito dev` is ever exposed beyond localhost.

### 11. [Low] Publish gate enforced on update but not on create

**Location:** create sets `published: body['published'] ?? false` at `packages/api/src/routes/admin/content.ts:388` with no publish-specific check; the update path gates publish transitions behind a `content:edit` check (`content.ts:512, 794`).

A `content:create`-only actor can create already-published content, sidestepping the stricter control on the update path. May be intentional (a prior commit notes `content:publish` "was never a real permission") — flagged for reconciliation against the intended role model.

**Remediation:** Decide whether publishing at create-time is allowed for `content:create`; if not, apply the same publish check the update path uses.

---

### 12. [Low] CDK worker policy not scoped to Manguito stacks

**Location:** `infra/lib/iam-worker-stack.ts:34-41` — `cloudformation:*` on `stack/*/*`.

The worker policy is otherwise well-scoped (resources pinned to `cdk-*`, `PassRole` conditioned to `cloudformation.amazonaws.com`, S3/ECR scoped to CDK asset resources — good least-privilege). The `cloudformation:*` on all stacks means a compromised worker credential can operate on any stack in the account (and CDK's default cfn-exec-role has broad rights). This is a known CDK tradeoff.

**Remediation:** Consider scoping to a Manguito stack name prefix, adding an MFA condition on the group, and attaching a permission boundary to bootstrap-created roles.

## What's solid (verified during the audit)

- **SQL injection:** `sql.raw` is only ever fed through strict identifier validators — `quoteIdent` (`^[a-z][a-z0-9_-]*$` + quote-escaping) and `quoteField` (`^[a-z][a-z0-9_]*$`). All values are parameterized. `sort_by` is whitelisted against `SORTABLE_FIELDS`. (`packages/api/src/repositories/content.ts:36-97,166-173`.)
- **Mass assignment:** admin content create builds `columnData` from a schema-field whitelist; the raw request body is never passed to the repository. (`content.ts:386-398`.)
- **Token revocation:** JWTs carry `token_version`; the auth middleware re-checks it against the DB on every request, and user edits / logout / password change bump it (`users.ts:91,306,419`, `auth.ts:210`) — so role changes invalidate live tokens. Proactive refresh is gated by the same version match.
- **Password storage:** `bcryptjs` with cost 12 (`packages/core/src/auth.ts:3`).
- **Cookies:** `httpOnly`, `secure`, `SameSite=Strict`; refresh token path-scoped to `/admin/api/auth` (`auth/jwt.ts:57-62`, `routes/admin/auth.ts:139`).
- **Secrets hygiene:** no secrets committed; `.env*` gitignored and untracked; infra secrets sourced from deployer env at synth.
- **Runtime S3 role:** scoped `bucket.grantWrite(taskRole/fn)` rather than `s3:*` (`fargate-stack.ts:83`, `lambda-stack.ts:69`).
- **Uploads traversal:** `/uploads/` handler strips `..` and enforces directory containment, sets `nosniff`, and forces `attachment` for unknown types (`server-entries.ts:110-124`).
- **JWT alg:** verification pins `HS256` explicitly, avoiding algorithm-confusion (`auth/jwt.ts:34`).

## Suggested remediation ordering

1. **#1 path traversal** and **#2 stored XSS** — user-facing, highest impact; both live in the generated server + upload serving path and can be fixed together.
2. **#7 secrets → Secrets Manager**, **#6 remove no-op shims**, **#4 CORS**, **#3 login timing**, **#5 spraying**, **#8 presigned content** — auth/infra hardening.
3. **#9 headers**, **#10 deps**, **#11 publish gate**, **#12 IAM scoping** — hardening / cleanup.

These can be turned into a remediation implementation plan (one task per finding, TDD with a regression test each) via the writing-plans workflow.
