# Deploying to AWS Lambda

This documents every obstacle encountered when first deploying the Manguito CMS
sandbox to AWS Lambda via `pnpm deploy:lambda`, and the fix applied for each.
Use it as a pre-flight checklist for subsequent deploys or when setting up a new
AWS account/region. It assumes [fargate.md](./fargate.md) has already been read
— several of the issues there (CDK bootstrap drift, Docker credential helper,
pnpm 10 build-script blocking, the `manguito` bin symlink ordering,
`ERR_PNPM_DEPLOY_NONINJECTED_WORKSPACE`, the `--require-approval never` flag)
apply identically here, since Lambda and Fargate build from the same root
`Dockerfile`.

---

## Prerequisites

- Everything in [fargate.md](./fargate.md)'s prerequisites — same AWS IAM
  policy, same `infra/.env`, same Docker + pnpm setup
- No additional prerequisites specific to Lambda

---

## The deploy command

```bash
cd infra
set -a && source .env && set +a
pnpm deploy:lambda
```

Which runs: `cdk deploy LambdaStack --require-approval never`

No separate build step is needed beforehand — the Docker build (triggered by
`cdk deploy`) builds the sandbox from source.

---

## Architecture overview

```
Browser (HTTPS)
    │
    ▼
Lambda Function URL (*.lambda-url.<region>.on.aws — AWS-managed TLS cert)
    │
    ▼
Lambda (container image) — dist/handler.handler
    │                │
    ▼                ▼
Neon Postgres    S3 bucket
(Neon HTTP driver, (IAM role grant,
 public internet)   no static creds)
```

No CloudFront needed here, unlike Fargate — the Function URL is HTTPS by
default, so the app's `Secure: true` auth cookie works without any extra
TLS-termination layer.

## Issues encountered and fixes applied

### 1. Zip-based Lambda exceeds the 250MB unzipped size limit

**Symptom**

Not a deploy-time error — this was caught during local testing, before ever
touching AWS. The original `LambdaStack` packaged just `apps/sandbox/dist/` as
a zip (`lambda.Code.fromAsset(path.join(__dirname, '../../apps/sandbox/dist'))`).
Running `pnpm --filter sandbox deploy --prod <dir>` to produce a self-contained
bundle (the same approach Fargate uses) and checking its size:

```bash
du -sh .lambda-bundle
# 298M
```

That's over Lambda's 250MB unzipped package limit.

**Root cause**

`dist/handler.js` imports `@bobbykim/manguito-cms-api` and
`@bobbykim/manguito-cms-db` directly rather than bundling them — same root
cause as Fargate needing a real `node_modules`. But `pnpm deploy --prod` under
`force-legacy-deploy=true` (required for pnpm 10, see fargate.md issue 5)
doesn't fully exclude devDependencies: `typescript`, `vitest`, `esbuild`
(three platform variants), `vite`, `drizzle-kit`, `jsdom`, and `msw` all ended
up in the bundle.

**Fix**

Switch to a **container-image Lambda** instead of a zip — no package-size
limit, and it reuses the exact multi-stage build already proven for Fargate.
Added a `lambda` target to the root `Dockerfile`, sharing the same `builder`
and `pruned` stages Fargate uses, but with a Lambda-compatible base image that
carries the AWS Lambda Runtime Interface Client:

```dockerfile
FROM node:22-alpine AS fargate
WORKDIR /app
COPY --from=pruned /app .
COPY --from=builder /repo/apps/sandbox/dist ./dist
EXPOSE 3000
CMD ["node", "dist/server.js"]

FROM public.ecr.aws/lambda/nodejs:22 AS lambda
WORKDIR /var/task
COPY --from=pruned /app .
COPY --from=builder /repo/apps/sandbox/dist ./dist
CMD ["dist/handler.handler"]
```

`LambdaStack` then uses `lambda.DockerImageFunction` +
`lambda.DockerImageCode.fromImageAsset(repoRoot, { file: 'Dockerfile', target: 'lambda' })`
instead of `lambda.Function` + `lambda.Code.fromAsset`. `FargateStack` was
updated to pass `target: 'fargate'` explicitly to its
`ecs.ContainerImage.fromAsset` call, since the Dockerfile now has two final
stages instead of one implicit default.

### 2. `Cannot find package 'hono' imported from /var/task/dist/handler.js`

**Symptom** (caught locally, via the [AWS Lambda Runtime Interface Emulator](https://github.com/aws/aws-lambda-runtime-interface-emulator), before deploying)

```json
{"errorType":"Error","errorMessage":"Cannot find package 'hono' imported from /var/task/dist/handler.js", ...}
```

**Root cause**

`dist/handler.js` (generated by `packages/cli/src/codegen/server-entries.ts`)
imports `hono/aws-lambda` directly:

```typescript
import { handle } from 'hono/aws-lambda'
```

But `hono` was only a *transitive* dependency — declared in
`packages/api/package.json`, not in `apps/sandbox/package.json`. pnpm's strict
linking doesn't expose transitive dependencies at a package's top level (no
phantom dependencies), so `hono` wasn't resolvable from `/var/task` even
though it physically existed somewhere in the pruned `node_modules` tree (one
level down, inside `@bobbykim/manguito-cms-api`'s own resolution scope).
`dist/server.js` never hit this because it only imports `@bobbykim/*` packages
and Node builtins — `hono` is used only inside the `api` package, never
imported directly by `server.ts`.

**Fix**

Add `hono` as a direct dependency of `apps/sandbox/package.json`, matching the
version range `packages/api` already uses:

```json
"dependencies": {
  "@bobbykim/manguito-cms-admin": "workspace:*",
  "@bobbykim/manguito-cms-api": "workspace:*",
  "@bobbykim/manguito-cms-cli": "workspace:*",
  "@bobbykim/manguito-cms-core": "workspace:*",
  "@bobbykim/manguito-cms-db": "workspace:*",
  "hono": "^4.12.0"
}
```

**Caught before any AWS deploy** by building the `lambda` Docker target
locally and invoking it with the Lambda Runtime Interface Emulator:

```bash
docker build --target lambda -t manguito-cms-lambda-test -f Dockerfile .
docker run -d --name lambda-test-run -p 9000:8080 \
  -e DB_URL=... -e AUTH_SECRET=... -e S3_BUCKET=... -e ALLOWED_ORIGIN=... \
  -e AWS_ACCESS_KEY_ID=... -e AWS_SECRET_ACCESS_KEY=... -e AWS_REGION=... \
  manguito-cms-lambda-test

curl -XPOST "http://localhost:9000/2015-03-31/functions/function/invocations" \
  -d '{"version":"2.0","rawPath":"/api/openapi.json","headers":{"host":"localhost"},"requestContext":{"http":{"method":"GET","path":"/api/openapi.json"}},"isBase64Encoded":false}'
```

This local loop (build → run → invoke → check logs) is much faster than a full
`cdk deploy` round-trip and is worth running before every Lambda deploy that
touches dependencies or the handler entry point.

### 3. `/admin` returns 404 (admin panel doesn't load) — but `/api/*` works fine

**Symptom**

After a successful deploy, `GET /api/openapi.json` and `GET /api/blog` both
return `200`, but `GET /admin` returns `404 Not found`. The exact same build
worked fine on Fargate.

**Root cause**

Fargate and Lambda run **two different generated entry point files**:

- Fargate's container runs `CMD ["node", "dist/server.js"]` — `server.ts`'s
  generated code is a hand-rolled raw Node `http` server with its own,
  separate static-file-serving logic: it intercepts any `/admin/*` request,
  serves the matching file from `dist/admin/` if one exists, and falls back to
  `dist/admin/index.html` otherwise (SPA history-mode routing). This logic
  lives entirely outside the Hono `app`.
- Lambda's container runs `CMD ["dist/handler.handler"]` — `handler.ts`'s
  generated code is just `handle(app)`, where `app` is the Hono instance from
  `createCmsApp()`. That Hono instance only ever has routes for `/api/*` and
  `/admin/api/*` (the real REST/admin API) — nothing serves the built admin
  SPA's static files. Any request to `/admin` falls through Hono's router with
  no match, producing a 404.

The admin-static-serving feature had only ever been written into `server.ts`;
`handler.ts` (and `vercel.ts`, which has the identical gap) never received it.

**Fix**

Added `adminStaticRoute()` to `packages/cli/src/codegen/server-entries.ts`,
registering the same static-file + `index.html` SPA-fallback behavior as a
Hono route, appended to `app` after `createCmsApp()` runs:

```typescript
app.get(`${ADMIN_PREFIX}/*`, async (c) => {
  const rel = c.req.path.slice(ADMIN_PREFIX.length) || '/'
  const candidate = resolve(adminDir, '.' + rel)
  let filePath = candidate
  try {
    const s = await stat(candidate)
    if (!s.isFile()) filePath = resolve(adminDir, 'index.html')
  } catch {
    filePath = resolve(adminDir, 'index.html')
  }
  try {
    const data = await readFile(filePath)
    return new Response(data, {
      headers: { 'Content-Type': ADMIN_MIME[extname(filePath).slice(1).toLowerCase()] ?? 'application/octet-stream' },
    })
  } catch {
    return c.text('Not found', 404)
  }
})
```

Generated into both `handler.ts` and `vercel.ts` (same gap exists for Vercel,
fixed pre-emptively).

This relies on Hono resolving a literal route over a wildcard registered
later, regardless of order — confirmed with a small standalone test:
`/admin/api/auth/login` (registered first) still matches the specific route
even after `/admin/*` (registered second) is added, so the admin API/auth
routes are unaffected by the new wildcard.

Verified locally with the Lambda Runtime Interface Emulator before
redeploying: `/admin` → 200 (SPA shell), `/admin/some/deep/route` → 200 (SPA
fallback), `/admin/api/config` → 401 (still hits the real, auth-protected
admin API, not the static-file wildcard).

## Files changed from the baseline

| File | Change |
|------|--------|
| `Dockerfile` | Added a `lambda` build target (Lambda base image, `CMD ["dist/handler.handler"]`); renamed the original final stage to `fargate` |
| `infra/lib/fargate-stack.ts` | Added `target: 'fargate'` to `ecs.ContainerImage.fromAsset` (Dockerfile now has two final stages) |
| `infra/lib/lambda-stack.ts` | Switched from `lambda.Function` + `lambda.Code.fromAsset(dist)` to `lambda.DockerImageFunction` + `lambda.DockerImageCode.fromImageAsset(repoRoot, { file: 'Dockerfile', target: 'lambda' })` |
| `infra/package.json` | Removed the now-redundant `pnpm --filter sandbox run build` pre-step from `deploy:lambda` — the Docker build does it |
| `apps/sandbox/package.json` | Added `hono` as a direct dependency |
| `packages/cli/src/codegen/server-entries.ts` | Added `adminStaticRoute()`, generated into both `handler.ts` and `vercel.ts` |

## Final verification

```bash
# Health check — should return 200
curl -s -o /dev/null -w "%{http_code}" \
  https://<function-url>/api/openapi.json

# Auth guard — should return UNAUTHORIZED JSON (server running, auth working)
curl -s https://<function-url>/admin/api/config

# Admin panel — should serve the Vue SPA (HTML response)
curl -s -o /dev/null -w "%{http_code}" \
  https://<function-url>/admin
```

The Function URL is printed as `LambdaStack.FunctionUrl` in the CDK deploy
output.
